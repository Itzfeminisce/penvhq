# v0.8 plan ‚Äî ambient delivery: the schema is the pin list

> **Status: active.** Decided 2026-07-19, in deliberation over a real wound:
> penv-cloud ‚Äî penv's own hosted product, dogfooding `@env` strictly ‚Äî had to
> hand-write `bridge-values.ts`, a file that maps `@env` values onto the exact
> `process.env` keys the WorkOS SDK reads, plus a hand-kept list of ambient keys
> to delete so nothing configures the SDK behind `@env`'s back. That is the
> local‚Üîconsumer translation seam penv exists to delete, rebuilt by hand inside
> penv's own product. This milestone deletes it. The [RFC](./RFC.md) owns *why*
> (see "The ambient surface is a projection, and the schema is the pin list");
> this file owns *how*, and in what order.

**One framing to carry through.** Three kinds of code read configuration: your
own code (`import { env } from "@env"` ‚Äî solved), third-party SDKs that read
`process.env.THEIR_EXACT_NAME` at module load (WorkOS, Prisma, NextAuth ‚Äî the
wound), and client bundles the framework inlines at build time (`NEXT_PUBLIC_*`,
`VITE_*` ‚Äî half-solved via the `next.config.ts` import). v0.8 makes all three
read the same truth with zero per-SDK code, using **no new vocabulary**: the
concept is v0.7's *projection* delivered into the local process instead of into
GitHub, and every declaration involved already exists ‚Äî the schema, `override`
(n√© `names`), and an import penv already ships.

---

## The settled surface

1. **`import "@penvhq/penv/config"` is promoted from compat-only to blessed.**
   Same import, three fixes:
   - It runs `load()` first: the mirror writes only validated values, so an SDK
     never sees a half-configured surface ‚Äî a missing required parameter fails at
     boot with penv's error naming the parameter, not the SDK's mystery three
     requests in.
   - It writes each parameter under its generated variable (`override` bending
     names exactly as `generate` and `push` already do ‚Äî one transform, three
     consumers).
   - **Exclusivity ‚Äî the schema is the pin list.** Every parameter the schema
     declares is penv's ambiently: written to `process.env` when it resolves to a
     value, **deleted** when it does not. Declare `workos/api-hostname` as
     `.optional()` and an ambient `WORKOS_API_HOSTNAME` can never steer the SDK
     behind `@env`'s back; leave it undeclared and penv does not touch it. The
     declaration the developer already makes is the whole contract ‚Äî no second
     list, which is what kills the `PINNED_OPTIONAL_KEYS` class of hand-kept
     files.

2. **The ordering race is solved at the framework's own pre-app seam, and
   `detect` owns which seam that is.** The import only helps if it runs before
   the SDK module reads `process.env`, and module-graph order is not something a
   developer controls. So `penv init` scaffolds the import into the one place
   each framework *guarantees* runs first ‚Äî detection as an input to `init`,
   never a stored identity (invariant 22), exactly as `schemaFile` and
   `publicPrefixes` already work:

   | Framework | Server runtime seam | Build-time (client vars) seam |
   |---|---|---|
   | Next.js | `instrumentation.ts` | `next.config.ts` |
   | Nuxt / Nitro | a Nitro plugin (`server/plugins/penv.ts`) | `nuxt.config.ts` |
   | TanStack Start | its server entry | `vite.config.ts` |
   | Vite SPA (no server) | ‚Äî (does not exist) | `vite.config.ts` |
   | SvelteKit | `hooks.server.ts` | `vite.config.ts` |
   | Plain Node / Express / Fastify | `node --import` in the start script, or line 1 of the entry | ‚Äî |
   | Bun | `preload` in `bunfig.toml` | ‚Äî |

   Two honesty rules ride with the table. A client-only app has no runtime half,
   so `init` scaffolds only the build half and says so ‚Äî which halves *exist* is
   part of detection. An unrecognised framework is asked, never guessed: `init`
   prints the import and the `node --import` form and lets the developer place
   it knowingly. And the table above is *planning* data ‚Äî each scaffold is
   implemented against the framework's current documentation at build time, not
   from this file's memory.

3. **`names` becomes `override`, with typed keys.** The block maps a parameter
   to the exact variable a consumer expects ‚Äî `override` is what it does. Keys
   are typed by the same declaration-merging mechanism the providers use: the
   scaffolded `.penv/env.ts` registers the schema's inferred shape on core's
   `PenvSchemaShape` (a *type-only* `declare module`, erased at runtime, so no
   config‚Üîschema cycle), and `override`'s key type narrows to the parameter ids
   the schema declares ‚Äî camelCase kebab-cased, mirroring the runtime transform.

   ```ts
   // .penv/env.ts ‚Äî penv init writes this block
   declare module "@penvhq/core" {
     interface PenvSchemaShape {
       readonly shape: z.infer<typeof schema>;
     }
   }
   ```

   ```ts
   // penv.config.ts ‚Äî override keys now autocomplete; a typo is a compile error
   override: { "workos/redirect-uri": "NEXT_PUBLIC_WORKOS_REDIRECT_URI" },
   ```

   No registration means the keys stay the open `string` type ‚Äî typing is a
   reward for the one scaffolded line, never a requirement. (Shipped early with
   the rename PR rather than held for v0.8; the exported `OverrideKeysOf<T>`
   transform lets a project opt in by hand too.)

4. **`doctor` learns `ambient-shadow`.** An ambient `process.env` key that
   collides with a schema-declared parameter's variable *outside* the mirror's
   control is a warning (something else is configuring the surface penv owns); a
   declared-and-deleted one is a pass; an exposed parameter that does not
   resolve is already `missing`'s failure. Firing and quiet tests, per the
   doctor rule.

## What the developer does

```bash
npx penv init        # scaffolds the seam files for the detected framework
```

Then only ever: declare parameters in the schema, `penv set` values, and add an
`override` entry when an SDK demands a name penv would not generate. Every way
of reading configuration ‚Äî `@env`, `process.env`, the client bundle ‚Äî sees the
same validated truth.

## Build order

1. **PR: `names` ‚Üí `override` + typed keys.** Small, breaking, rides the 0.5.0
   release train (land before the Version Packages PR merges). `PROVIDER_TYPE_LEGACY`-style
   migration error for a config still saying `names`.
2. **PR: the blessed mirror.** `@penvhq/penv/config` gains validate-first and
   schema-exclusivity; `ambient-shadow` in doctor; Documentation.md's compat
   caveat replaced by the blessed story.
3. **PR: the seams.** `init`/`detect` scaffold per framework, halves only where
   they exist, ask-don't-guess fallback.

**Gate.** penv-cloud deletes `bridge-values.ts` and `env-bridge.ts`, moves the
WorkOS surface (required and optional keys alike) into its schema plus one
`override` entry, runs the scaffolded seams ‚Äî and WorkOS authenticates with no
penv-aware code anywhere in the app. The check that proves exclusivity: export a
stray `WORKOS_API_HOSTNAME`, boot, and confirm the SDK never sees it.

## Deliberately deferred

- **A lazy `process.env` proxy** (ordering independence with no seam file at
  all). Real magic, real platform risk ‚Äî edge runtimes and env-snapshotting
  hosts break it ‚Äî so it stays a future possibility until the eager mirror
  proves the semantics.
- **Selective exposure** (mirroring less than the whole schema). The dotenv
  world has always had every value ambient in-process; if a team asks for a
  narrower surface, that is a deliberation, not a hedge shipped now.

## What this plan deliberately does not do

- **It does not touch the runtime's shape.** `load()` stays synchronous and
  disk-backed; the mirror is a delivery step downstream of resolution, on the
  same side of the design as `generate`.
- **It does not add a `framework` key to config** ‚Äî invariant 22 stands.
  Detection happens at `init`, what lands is the scaffolded files.
- **It does not make penv ambient by default.** No import, no mirror: a library
  consumer who never asked for `process.env` writes gets none.
