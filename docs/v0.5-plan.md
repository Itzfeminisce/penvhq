# v0.5 implementation plan ‚Äî provider proof (Vault) and rotation

This is the working plan for the pivotal milestone. It is scoped from the code as it stands on `feat/v0.1-v0.2-filesystem-core`, not from the design in the abstract. The [roadmap](./Roadmap.md#v05--provider-proof-the-pivotal-milestone-and-rotation) owns *whether* and *when* v0.5 is resourced ‚Äî including the open market question it gates on ‚Äî and the [RFC](./RFC.md) owns *why* the shapes below are what they are. This file owns *how*, and only that.

**One framing to carry through.** Building "a Vault adapter" is mostly not about Vault. The Vault client is the reward at the end; the risk lives in four pieces of shared machinery that do not exist yet, three of which change code every provider will share. The v0.5 gate ‚Äî *flip an environment from `filesystem` to `vault` with zero application edits* ‚Äî is a test of that machinery, with Vault as the first thing plugged into it. Sequence the machinery first so Vault has somewhere to land.

---

## What is already true (and makes this smaller than it looks)

Three properties of the v0.1‚Äìv0.4 code mean the adapter itself is the easy 40%:

1. **Providers store opaque strings; they never encrypt.** Core seals a value into a single-line text envelope (`penv:1:<keyId>:<nonce>:<ciphertext>`, `packages/core/src/envelope.ts`) precisely so it round-trips through any store that holds a string. `.enc`, key sources, and seal/open all live in `@penvhq/core` and never reach the provider. The Vault adapter stores whatever string it is handed. Encryption is free.

2. **A provider is never on the runtime hot path.** Per the RFC's *a provider is a sync target, not a runtime source*, the application boots off the local `.penv` tree; `penv pull` materialises it. Vault is touched only by `pull`, `push`/`set`, and `doctor`. So every contract method being `async` costs nothing ‚Äî `runtime/src/resolve.ts` never awaits a provider, and must keep not doing so.

3. **The acceptance test already exists and is provider-agnostic.** `runProviderContractSuite(name, makeProvider)` in `packages/providers/filesystem/src/contract.ts` speaks only `@penvhq/core`'s vocabulary ‚Äî no paths, no `node:fs`. It *is* the "same behavioural suite" the gate names. A Vault package points `makeProvider` at a test mount and must go green with zero suite edits. Any edit to the suite to accommodate Vault is a design finding, not a fix (see ¬ß3).

## The contract, verbatim

`packages/core/src/types.ts:180`. Seven async methods over `ValueFile` / `ParameterRef`, absence treated as success on every read and remove:

```ts
interface Provider {
  readonly type: string;
  read(file: ValueFile): Promise<string | undefined>;
  write(file: ValueFile, value: string): Promise<void>;
  list(): Promise<ValueFile[]>;
  remove(file: ValueFile): Promise<void>;
  readMeta(ref: ParameterRef): Promise<Meta | undefined>;
  writeMeta(ref: ParameterRef, meta: Meta): Promise<void>;
  removeMeta(ref: ParameterRef): Promise<void>;
}
```

The unit of address is `ValueFile` ‚Äî a `ParameterRef` (`namespace: string[]`, `name`) plus a `Scope` (the 4-member cascade union) plus `encrypted: boolean`. Meta is addressed by the bare `ParameterRef` and is always plaintext JSON. The Vault adapter's whole job is to turn those into KV v2 paths and back.

## The four gaps, in build order

Ordered so each step unblocks the next and the risky shared-code changes land before the Vault dependency does. Steps 1‚Äì3 have no Vault dependency at all.

### Step 1 ‚Äî Provider registry and dispatch (pure refactor, no Vault)

**The gap.** There is no registry. Both consumers hardcode the filesystem:
- `packages/cli/src/project.ts` ‚Äî `openProject` instantiates `createFilesystemProvider(...)` and types `Project.provider` as the *concrete* `FilesystemProvider`, not the `Provider` interface.
- `packages/runtime/src/resolve.ts` ‚Äî imports `createFilesystemProvider` directly (and correctly never inspects `providers.*.type`, because the runtime reads disk regardless of provider ‚Äî that invariant stays).

**The work.**
- Introduce a registry keyed by `ProviderConfig.type` ‚Üí factory, living in the CLI (not core, and not runtime). Core defines the `Provider` interface; it does not know provider *implementations* exist.
- Retype `Project.provider` to the `Provider` interface. Audit every `Project.provider` callsite for reliance on `FilesystemProvider`-only surface (the sync twins, `get root()`), and confine those to filesystem-specific paths.
- Leave `runtime/src/resolve.ts` untouched in behaviour: it keeps reading the local tree. Add a test that pins the runtime never imports a provider registry ‚Äî the runtime must not gain the ability to select a network provider.

**Why first.** This is the portability seam. The v0.5 gate is a test *of this step*. It carries no Vault dependency, so it can land, be reviewed, and be proven against the filesystem provider alone ‚Äî flipping a config `type` with only the filesystem registered should still resolve, and a config naming an unregistered type should fail with a clear error at `openProject`, not a crash later.

**Gate for the step.** With only `filesystem` registered, the full existing suite passes unchanged, `Project.provider` is the interface type, and an unknown `providers.*.type` is refused at config-open time.

### Step 2 ‚Äî Extend the contract with `readPrevious` as a declared, optional capability

**The gap.** The `Provider` interface has no retention operation. Rotation fields exist today only as pass-through `Meta` keys the suite uses to prove unknown keys survive.

**The work ‚Äî and it is a contract change, handled as one.** Per the RFC's *previous-value retention is a declared capability, not a universal verb*, the shape is already settled and must not be re-litigated during the build:
- Add an **optional** capability, not a mandatory method. A provider *declares* whether it retains; the shape is "give me the previous value" and nothing more ‚Äî no TTL, no retention policy, because those are not portable (Vault has a time TTL, SSM a fixed count of 100, Kubernetes nothing).
- `dual-valid` rotation requires a retaining provider; `atomic-cutover` does not; the filesystem and (later) Kubernetes legitimately declare no retention.
- penv owns the grace-window clock (`rotatingSince` in meta). A provider is never asked to enforce a window ‚Äî it is only asked, and must be prepared to answer *no*, because provider pruning and penv's timer do not commute. A rotation that cannot find its previous value is a `doctor` failure, never a silent success ‚Äî the same rule the encryption check already follows for a sealed value penv cannot open.

**Why before the adapter.** The adapter implements this capability; it cannot be written against an interface that lacks it. Doing it as a deliberate contract edit now is the RFC rule working on schedule ‚Äî a contract change is a finding taken up front, not an edit discovered mid-adapter.

### Step 3 ‚Äî Relocate the contract suite so a sibling provider can consume it

**The gap.** `runProviderContractSuite` lives inside `packages/providers/filesystem/src/contract.ts` and is deliberately **not** re-exported from that package's `index.ts` ‚Äî it imports `vitest` at module scope, so it must never enter a consumer's runtime.

**The work.** Move it to a shared test-only package (`@penvhq/provider-contract`, or similar) that both the filesystem and Vault packages import from their `*.test.ts` files. v0.6 (SSM, Kubernetes) will reuse it too, so relocation now avoids a subpath-import hack repeated three times.

**Constraint.** The suite stays filesystem-free and must keep asserting only over contract vocabulary ‚Äî no `list()` ordering, no on-disk layout. If Vault forces a change to the suite, that is a finding about the contract, surfaced and decided, not a quiet accommodation.

### Step 4 ‚Äî The Vault adapter (`@penvhq/provider-vault`)

Only now, with a seam to plug into, a capability to implement, and a suite to run against.

**Package.** `packages/providers/vault` as `@penvhq/provider-vault`, matched by the existing `packages/providers/*` workspace glob (no workspace edit). Mirror the filesystem package's scaffolding: `type: module`, dual tsup output, `@penvhq/core` as a `workspace:*` dependency, `VaultProvider implements Provider` + a `createVaultProvider(options)` factory, an `index.ts` barrel that excludes the test suite.

**The four sharp edges the RFC already named:**

1. **KV v2 required and validated, not assumed.** KV v1 overwrites with no history ‚Äî "any update will overwrite the original value and not recoverable" ‚Äî so a v1 mount is an adapter that silently cannot rotate. Check the mount's `options.version == 2` at config time and refuse v1 there, not mid-rotation.

2. **Meta at its own provider address**, never Vault's `custom_metadata`. That facility is a flat `map<string,string>` capped at 512-byte values; penv's meta is nested and passes 512 bytes with one honest `description` plus two environment blocks. Meta is a sibling record at its own KV path ‚Äî exactly what the filesystem does with the sibling `.json`. `list()` must exclude meta addresses (the suite asserts a written meta is never returned as a value).

3. **`list()` is a recursive walk.** Vault's LIST returns one level ‚Äî "list on a file will not return a value" ‚Äî so enumerating a tree is N round-trips, not one scan. This mirrors the filesystem's `#walk`, over the network.

4. **Explicit `(env, namespace, name) ‚Üí Vault path` mapping**, driven by `providers.*.path`. That config field exists in `ProviderConfig` today and is consumed by nothing; this is its first real use. The mapping is explicit config, not inference.

**Transport decision (to settle at the top of this step).** Two options: talk to Vault over its HTTP API (a real dependency, testable against a dev-mode Vault in CI), or shell out the way `@penvhq/sink-github` shells to `gh`. The sink precedent leans toward a thin boundary that keeps credentials out of penv's process. Decide before writing the client; do not build both.

**`readPrevious` for Vault.** `?version=N` addresses a prior value. Implement the capability declared in Step 2, and prove it is prepared to be told the version was pruned (Vault's `max_versions` defaults to 10).

## Rotation, `pull`, and the mock provider (ride with v0.5)

These are milestone-scope alongside the adapter, not the adapter itself:

- **`penv pull`** ‚Äî materialise the local tree from the provider. Only meaningful once a real provider exists. It is penv's version of a step most deploys already have (the Vault Agent Injector, the CSI driver, External Secrets all write files).
- **Rotation** ‚Äî `dual-valid` and `atomic-cutover` as distinct mechanisms; the `rotatingSince` clock distinct from `lastRotated`; `doctor` overdue (`now - lastRotated > policy`) and stuck (`now - rotatingSince > threshold`, gated to `dual-valid` so it does not false-positive on atomic-cutover passwords) checks.
- **The mock provider** ‚Äî for rehearsing rotation locally, since the filesystem cannot exercise true dual-valid rotation (dual validity is a property of a live system, not a filesystem quirk ‚Äî Kubernetes sits on the same side of that line).
- **`penv doctor` cross-provider drift** against a live Vault ‚Äî value by value, because this provider can be read back, unlike a sink where value drift is always `unknown`.
- **Extract and document the provider contract** as the interface future adapters satisfy, with retention marked optional. This is the artifact v0.6 (SSM, Kubernetes) and v1.0 (public SDK) build against.

## The milestone gate (from the roadmap, restated as acceptance)

1. The Vault adapter passes the *same* `runProviderContractSuite` the filesystem provider passes ‚Äî no suite edits.
2. Flipping an environment from `filesystem` to `vault` in `penv.config.ts` requires zero application-code changes. Only after this passes is provider portability stated as proven rather than promised.
3. A `dual-valid` rotation runs `active ‚Üí rotating ‚Üí active` with grace-window overlap; `atomic-cutover` flips with no penv-layer grace window; `doctor` flags a stuck external rotation without false-positiving on atomic-cutover.
4. A KV v1 mount is refused at config time.

## What this plan deliberately does not do

- **It does not resolve the market question the roadmap gates v0.5 on** ‚Äî *where do the trial teams' production applications actually read their secrets from?* If Actions injects secrets at deploy and that is the whole story, this milestone aims at a market still to be found. That question is answered before resourcing, not inside this plan. This file is what to build *if* the answer says build it.
- **It does not touch SSM or Kubernetes.** They are v0.6, built against the contract this milestone extracts, with no contract changes permitted. The one bend already found ‚Äî retention is not portable ‚Äî is priced into Step 2 so it does not reopen v0.5 after the fact.
- **It does not add async to the runtime.** The runtime stays synchronous and disk-backed. Any change that makes `load` await a provider is out of scope and against the design.
